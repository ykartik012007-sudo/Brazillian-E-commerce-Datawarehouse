# 📘 Naming Convention

This document defines the naming standards used throughout the **Brazilian E-commerce Data Warehouse** project. Consistent naming improves readability, maintainability, and collaboration.

---

# 🌳 Project Naming Standards

## General Rules

- Use **snake_case** for all object names.
- Use **lowercase letters only**.
- Separate words using an underscore (`_`).
- Avoid spaces, special characters, and abbreviations unless they are widely accepted.

### ✅ Example

```text
customer_table
order_items
product_category
```

---

# 🥉 Bronze Layer

The Bronze layer stores raw data exactly as received from the source system.

## Table Naming Convention

```
<source>_<original_table_name>
```

### Rules

- Keep the original table name.
- Prefix the table with the source system.
- Store under the **bronze** schema.

### Examples

| Source Table | Bronze Table |
|--------------|--------------|
| olist_customers_dataset | `bronze.archive_olist_customers_dataset` |
| olist_orders_dataset | `bronze.archive_olist_orders_dataset` |
| olist_products_dataset | `bronze.archive_olist_products_dataset` |

### Examples

| Bronze Table | Silver Table |

---

# 🥇 Gold Layer

The Gold layer contains business-ready dimensional models.

## Table Naming Convention

```
<category>_<table_name>
```

### Categories

| Prefix | Description |
|---------|-------------|
| `dim` | Dimension Table |
| `fact` | Fact Table |

### Examples

| Silver Table | Gold Table |
|--------------|------------|
| `silver.archive_olist_customers_dataset` | `gold.dim_customers` |
| `silver.archive_olist_products_dataset` | `gold.dim_products` |
| `silver.archive_olist_orders_dataset` | `gold.fact_orders` |

---

# 📑 Column Naming Convention

## Surrogate Keys

Every dimension table must use a surrogate primary key.

### Pattern

```
<table_name>_key
```

### Rules

- Use the table name followed by `_key`.
- Surrogate keys should be integer values generated by the warehouse.

### Examples

```text
customer_key
product_key
seller_key
date_key
```

---

## Technical Columns

Technical (metadata) columns must start with the prefix:

```
dwh_
```

### Pattern

```
dwh_<column_name>
```

### Purpose

These columns store warehouse metadata such as:

- Load date
- Update date
- Source system
- Batch ID
- Record status

### Examples

```text
dwh_load_date
dwh_update_date
dwh_batch_id
dwh_source_system
```

---

# ⚙️ Stored Procedure Naming Convention

Stored procedures responsible for loading data into each layer must follow a consistent naming pattern.

## Pattern

```
load_<layer>
```

### Examples

```text
load_bronze
load_silver
load_gold
```
